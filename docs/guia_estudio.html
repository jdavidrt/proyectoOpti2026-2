1. Fundamentos — ¿Por qué programación entera?

Muchos problemas del mundo real requieren decisiones discretas: ¿abre o no abre este almacén? ¿entra o no entra este ítem en esta caja? Las variables continuas de la programación lineal (LP) no capturan esa realidad. La Programación Lineal Entera (ILP) exige que algunas o todas las variables sean enteras —generalmente binarias (0 o 1).

LP (relajada)

Permite fracciones. Puede decir "pon 0.3 del ítem en la caja 2". Rápida, convexa, pero irreal para decisiones binarias.

ILP (entera)

Solo 0 o 1. Captura decisiones binarias reales. NP-difícil en general, pero manejable con solvers modernos como Gurobi.

El método Branch-and-Bound (B&B)

El corazón de cualquier solver de ILP. En Gurobi, todo ocurre dentro de model.optimize() bin_packing.py L101:

  1. Relajar: resuelve el LP continuo (ignora integralidad). Obtiene cota inferior (lower bound).
  2. Verificar: si la solución ya es entera, terminó. Si no, elige una variable fraccionaria, e.g. \(x=0.7\).
  3. Ramificar: crea dos subproblemas — uno con \(x \leq 0\) y otro con \(x \geq 1\).
  4. Podar: si un nodo da cota inferior peor que la mejor solución entera conocida, descártalo.
  5. Repetir hasta agotar el árbol o alcanzar el gap deseado (en nuestro caso, 0 %).
El insight clave: cuanto más fuerte (más ajustada) sea la relajación LP, menos nodos necesita explorar el B&B. Si la relajación ya es entera, el solver termina en el nodo raíz.

¿Qué es el MIP Gap?

\[ \text{MIP Gap} = \frac{|\text{Mejor entero} - \text{Cota inferior}|}{|\text{Mejor entero}|} \times 100\% \]

En Gurobi se lee con model.MIPGap bin_packing.py L111. Un gap de 0.0000 % certifica la optimalidad. Ambos modelos lograron este resultado.

2. Bin Packing Problem (BPP)

2.1 La intuición

Tienes cajas de capacidad \(C = 150\) kg y 120 objetos de distintos pesos. Quieres meterlos todos usando la menor cantidad de cajas posible. Cada objeto va en exactamente una caja; ninguna caja puede sobrecargarse.

Instancia: Falkenauer u120_01 · 120 ítems · capacidad 150 · pesos ∈ [20,100] · archivo data/BinPacking_u120_01.txt · óptimo documentado: 48 contenedores (se corrieron 40 ítems por límite de licencia).

2.2 Formulación matemática

Conjuntos y parámetros

2.3 El límite de licencia: ITEM_LIMIT

La licencia gratuita (pip) de Gurobi acepta máximo ~2000 variables. El BPP completo necesita \(120^2 + 120 = 14{,}520\) variables. La solución: reducir a 40 ítems → \(40^2 + 40 = 1{,}640\) variables.

models/bin_packing.py — Interruptor de licencia L169
158# TRIAL license -> ITEM_LIMIT = 40 (40*40 + 40 = 1,640 vars < 2000)
159# ACADEMIC license -> ITEM_LIMIT = None (full 120-item instance)
169ITEM_LIMIT = 40← cambiar a None con licencia académica

2.4 La cota inferior de área

\[ \text{LB} = \left\lceil \frac{\sum_{i \in I} w_i}{C} \right\rceil = \left\lceil \frac{2151}{150} \right\rceil = \lceil 14.34 \rceil = \mathbf{15} \]

El solver reporta exactamente 14.34 como relajación LP y cierra en 15. El óptimo iguala la cota — el empaque es tan ajustado como físicamente posible.

Resultado: 40 ítems → 15 contenedores · gap 0.0000 % · tiempo 0.09 s · ver results/bin_packing_solution.txt

3. Set Covering Problem (SCP)

3.1 La intuición

Tienes 50 barrios y 200 posibles ubicaciones para estaciones de bomberos. Cada ubicación cubre ciertos barrios y tiene un costo. Quieres el subconjunto de estaciones de menor costo total que deje todos los barrios cubiertos.

Instancia: 50 filas × 200 columnas · densidad ≈ 21 % (2091 incidencias) · archivo data/SetCovering_50x200.txt · instancia completa (sin restricción de licencia).

4.2 ¿Qué hace x[j].X > 0.5?

Tras la optimización, Gurobi almacena el valor de cada variable en .X. Aunque las variables son declaradas binarias, los solvers internamente trabajan con punto flotante, por lo que una variable "1" puede ser 0.9999. El umbral > 0.5 es el criterio estándar para interpretar una variable binaria como 1:

models/set_covering.py L96 — Extraer columnas seleccionadas
96selected = [j for j in J if x[j].X > 0.5]← .X = valor óptimo de la variable
models/bin_packing.py L141–143 — Extraer asignaciones
141items_in_bin = [i for i in I if x[i, j].X > 0.5]← ítems asignados al bin j
142load = sum(w[i] for i in items_in_bin) # carga total del bin
143f.write(f" Bin {j:3d}: items={items_in_bin}, load={load}/{C}\n")

5. Literatura Canónica

BPP — Falkenauer (1996)

Referencia: Falkenauer, E. (1996). A hybrid grouping genetic algorithm for bin packing. Journal of Heuristics, 2(1), 5–30.

  • Introduce el Grouping Genetic Algorithm (GGA): opera sobre grupos (contenedores) como unidad, no sobre asignaciones individuales.
  • Define la instancia estándar u120 usada en este proyecto (data/BinPacking_u120_01.txt): 120 ítems, cap 150, óptimo 48.
  • GGA supera heurísticas clásicas como First Fit Decreasing (FFD).
  • La simetría del BPP (reordenar contenedores = misma solución) es la barrera principal para métodos exactos.

SCP — Beasley (1987)

Referencia: Beasley, J. E. (1987). An algorithm for set covering problem. European Journal of Operational Research, 31(1), 85–93.

  • Combina relajación Lagrangiana + subgradiente para cotas duales, + búsqueda local para factibilidad primal.
  • Crea las instancias de OR-Library incluyendo la 50×200 (data/SetCovering_50x200.txt).
  • Demuestra que la relajación Lagrangiana del SCP supera a la relajación LP estándar en instancias grandes.

7.2 Simetría como cuello de botella del BPP

Para 15 contenedores hay \(15! \approx 1.3 \times 10^{12}\) permutaciones equivalentes. La restricción y[j] >= y[j+1] en bin_packing.py L95 colapsa ese espacio a uno canónico.

Lección de modelado: una sola desigualdad de ordenamiento vale más que un mejor computador. El modelado bien hecho reemplaza al cómputo bruto.

7.3 Densidad del SCP

2091 incidencias en 50×200 = densidad ≈ 21 %. Solo 9 de 200 columnas cubren los 50 elementos: columnas "generalistas" de alta cobertura, costo total 42.

7.4 Implicaciones de dominio

BPP → Logística/manufactura

Cada contenedor evitado = un camión, caja o corte de lámina menos. Óptimo = cota de área → sin margen oculto: reducir contenedores requeriría rediseñar los objetos.

SCP → Localización/selección

9 recursos de costo 42 cubren 50 requisitos. Patrón: estaciones de emergencia, airline crew, feature selection en ML. LP fuerte → relajación rápida como guía confiable.

Contenedores vs. Capacidad

(a) Respuesta estructural — casi lineal

Al doblar C de 100 a 200, contenedores caen 23 → 11 (−52%), siguiendo de cerca \(\lceil 2151/C \rceil\). Meseta en C=180 y C=190 (ambos 12): los pesos indivisibles impiden aprovechar el espacio extra.

(b) Costo computacional — NO monótono

Pico en C=180: 2.27 s (≈75× la mediana). Ocurre cuando la solución entera queda al filo de la cota fraccionaria — el B&B debe descartar el contenedor de más. Los resultados se guardan en results/sensitivity_capacity.png y results/sensitivity_summary.csv — sensitivity_analysis.py L145–155.

9. Conclusiones

  1. Ambos modelos alcanzaron gap 0 %: SCP (costo 42, 9 columnas) y BPP (15 contenedores, 40 ítems).
  2. La fortaleza de la relajación LP explica el comportamiento computacional: LP=IP en SCP (1 nodo); LP<IP en BPP (depende de B&B).
  3. Ruptura de simetría y[j] >= y[j+1] fue decisiva en el BPP: evita explorar 15! permutaciones equivalentes con una sola línea de código bin_packing.py L95.
  4. Análisis de sensibilidad — dos capas: estructural (−52 % contenedores al doblar C, mesetas) y computacional (picos hasta 75× la mediana en capacidades de frontera).
  5. Modelado > cómputo bruto: formulación fuerte + ruptura de simetría supera a hardware más potente.
